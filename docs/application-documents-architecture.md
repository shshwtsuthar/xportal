### Application Documents, Offer Letters, and Storage Architecture (Planning)

This note defines how we will store, secure, and surface application-related documents (passport scans, letters, offer letters) across the New Application → Offer → Payment → Approval/Enrolment flow.

---

## Goals

- Centralize all application documents in Supabase Storage with strict, path-scoped RLS.
- Provide a simple dropzone in the wizard to upload/view/delete documents with progress and accessibility.
- Generate Offer Letter PDFs into the same storage hierarchy.
- Maintain an auditable metadata record for every file and link them to `applications` (and later to `clients/enrolments`).
- Keep permissions principle-of-least-privilege; avoid public buckets.

---

## Buckets and Foldering

- Bucket: `rto-docs` (private)
- Paths during application phase (pre-enrolment):
  - `applications/{applicationId}/uploads/{filename}`
  - `applications/{applicationId}/offer-letter/{offerId or timestamp}.pdf`
- After approval (optional normalization step):
  - Keep originals under `applications/{applicationId}/...` for immutable audit trail.
  - Optionally duplicate curated, durable documents under:
    - `students/{clientId}/enrolments/{enrolmentId}/` (derived artifacts)

Notes:
- Paths are server-enforced; the client never selects the final key.
- Offer letter artifacts always live under the application’s path for reproducibility and audit.

---

## Database Metadata Table

Add `sms_op.application_documents` to track each object and enforce RLS joins:

- `id uuid PK DEFAULT gen_random_uuid()`
- `application_id uuid NOT NULL REFERENCES sms_op.applications(id) ON DELETE CASCADE`
- `storage_key text NOT NULL`  // e.g., `applications/{id}/uploads/passport.pdf`
- `category text NOT NULL CHECK (category IN ('passport','letter','evidence','offer_letter','other'))`
- `filename text NOT NULL`
- `content_type text`
- `size_bytes bigint`
- `uploaded_by_staff_id uuid REFERENCES sms_op.staff(id) ON DELETE SET NULL`
- `created_at timestamptz NOT NULL DEFAULT now()`

Indexes:
- `application_id` (btree)
- `(application_id, category)` (btree)

Why metadata:
- Fast listing and filtering without listing the bucket.
- Joins for authorization via application ownership/role.
- Attach semantic category for UX and automation.

---

## Security & RLS

Storage bucket policy (conceptual):
- Bucket private; all access via signed URLs generated by Edge Functions.
- Edge Function validates caller roles and that the requested key is under `applications/{applicationId}/...` that the user is allowed to access.

RLS for `sms_op.application_documents`:
- Enable RLS.
- Policies:
  - Read: admins, compliance, staff with access to the application; agents if assigned to the application; (optional) the applicant if you expose a portal later.
  - Insert/Delete: admins and staff with write permission on the application.
- Tie to existing session JWT claims and `security.users` linkage to staff/roles.

Auditability:
- Keep metadata rows immutable; deletes should mark a `deleted_at` timestamp or hard-delete only for erroneous uploads (decision: start with hard-delete for simplicity, can soften later).

## API Surfaces (OpenAPI additions)

Documents (scoped to application):
- `GET /applications/{applicationId}/documents` → list from metadata table.
- `POST /applications/{applicationId}/documents/upload-url` → returns short-lived signed URL + required headers; body includes `filename`, `contentType`, `category`.
- `POST /applications/{applicationId}/documents/confirm` → after client PUTs to signed URL, backend upserts metadata row with size/type/sha256 (if provided).
- `DELETE /applications/{applicationId}/documents/{documentId}` → deletes storage object and metadata row (if permitted).

Offer Letter:
- `POST /applications/{applicationId}/offer-letter` → generates PDF, uploads to `applications/{id}/offer-letter/{timestamp}.pdf`, creates metadata row with `category='offer_letter'`, updates `applications.offer_letter_url` and `status → OfferGenerated`.
- `POST /applications/{applicationId}/send-offer` → emails the stored PDF, captures email log, sets `status → OfferSent` and `offer_sent_at`.

Approve (already planned):
- `POST /applications/{applicationId}/approve` → verifies cleared funds; creates enrolment; if international, creates CoE (PRISMS) and stores CoE artifacts under application path; sets `status → EnrolmentCreated`.

Error model:
- 401/403 for unauthorized.
- 409 `PaymentRequired` when approving without sufficient funds.
- 412 `InvalidState` for invalid transitions.

## Frontend Integration (Wizard)

Dropzone (Step 1 header area):
- Accessible drag-and-drop with keyboard support.
- Accept list: `image/*, application/pdf, application/msword, application/vnd.openxmlformats-officedocument.wordprocessingml.document` (configurable)
- On file selection:
  1. Request upload URL (`POST /applications/{id}/documents/upload-url`).
  2. `PUT` file to signed URL with returned headers.
  3. Call `POST /applications/{id}/documents/confirm` with final size/type/hash.
  4. Refresh list via `GET /applications/{id}/documents`.
- Show list with: filename, category chip, size, uploadedAt, delete action (with confirm dialog), and download (via a one-time signed URL `GET` helper or the list endpoint returning a `downloadUrl`).
- Offer Letter appears in the same list as `category='offer_letter'` once generated.

UX notes:
- Progress bar per-file; cancelable uploads.
- Debounce list refetch; optimistic add on confirm.
- A11y: `role="button"`, `aria-label`, `tabIndex=0`, keyboard handlers for dropzone and actions.

## Implementation Steps (Backend)

1) Storage
- Create `rto-docs` bucket (private).
- Configure CORS for PUT from the web app origin (only signed URLs will allow object writes).

2) DB
- Create `sms_op.application_documents` (idempotent migration).
- Add columns to `sms_op.applications`: `offer_letter_url text`, `offer_generated_at timestamptz`, `offer_sent_at timestamptz`, `first_instalment_required numeric(10,2)`, `payment_received_at timestamptz`, `is_international boolean`.
- Extend `sms_op.applications.status` to add: `ApplicationSubmitted`, `OfferGenerated`, `OfferSent`, `AwaitingPayment`, `Approved`, `EnrolmentCreated`.

3) Edge Functions
- `applications/documents`: list, signed-upload, confirm, delete.
- `applications/offer-letter`: generate PDF, upload, record metadata + status.
- `applications/send-offer`: email with attachment from storage.
- Update `applications/approve`: payment gating, enrolment creation, PRISMS CoE for internationals, store CoE payload/artifacts.

4) RLS/Policies
- Enable RLS on metadata table; add role-based read/write policies.
- Storage access via signed URLs only; no public read.

## PRISMS and CRICOS Timing (Clarity)

- CoE issuance happens at approval for international students (after cleared funds) because PRISMS requires an official CoE before commencement/visa conditions, not at course completion.
- The `cricos.confirmations_of_enrolment` record is created upon CoE issuance (status `Issued`), not after the enrolment finishes. This mirrors legal reality and enables lifecycle events (e.g., cancellation) to be tracked.
